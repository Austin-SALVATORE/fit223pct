# Exercise Asset Pipeline

Canonical plan for turning generated exercise strips into runtime app assets.
Agreed 2026-07-19. Complements [ExercisePromptGenerationPlan.md](ExercisePromptGenerationPlan.md)
(prompt library + generation) and supersedes the format rules in
[ExerciseAssets.md](../ExerciseAssets.md) where they conflict (see
"Relationship to prior docs" below).

## QA policy

The owner's visual review is the only acceptance gate. Checklist-based QA
(prompt.md "Form checkpoints") is advisory input, never a veto. Assets the
owner placed or approved are never regenerated, overwritten, or deleted
without explicit instruction.

## Generation

- Model: `gpt-5.6-sol`, reasoning effort high, via OpenAI Responses API with
  the `image_generation` tool (`scripts/generate-via-sol.mjs`) — reproduces
  the manual ChatGPT-UI flow that produced the approved pilot images.
- Input per exercise: goblet-squat `reference.png` as style anchor + the
  fenced prompt from the exercise's `prompt.md`.
- Output: one multi-frame strip → `<id>/reference.png` (the source artifact).
- Retired: `gpt-image-1` (all architectures) — fails frame count and scene
  coherence. Do not use.

## Per-exercise asset contract

```text
public/assets/exercises/<id>/
  prompt.md            # generation source (committed)
  reference.png        # image source — precache-excluded, slicing input
  reference.avif       # runtime: full strip
  thumbnail.avif       # runtime: card/list image, from thumbnailFrame
  frames/01..NN.avif   # runtime: step-through frames (1-indexed, zero-padded)
  metadata.json        # authoring metadata (see below)
```

- **One shipped format: AVIF.** Chosen for gradient handling and size;
  validated during the pilot (compare WebP side-by-side once, then commit).
- **Hybrid model:** the strip is canonical; frames are derived by slicing.
  No flip-book animation — frames are positional milestones, not uniform
  time samples, and are not pixel-registered. True motion is a later era.
- Frame slicing: white-gutter (ink-density minima) detection with a
  minimum-gutter-width rule; manual cut overrides allowed per exercise.

## Metadata: authoring vs runtime

Authoring file, per folder, committed — carries only what the asset layer
uniquely knows. Domain data (muscles, equipment, cues, difficulty) stays in
`src/domain` / Dexie seed and is never duplicated here:

```json
{
  "exerciseId": "goblet-squat",
  "version": 1,
  "media": { "frameCount": 6, "thumbnailFrame": 4 },
  "credits": { "source": "gpt-5.6-sol via OpenAI API", "generated": "2026-07-19" }
}
```

Runtime manifest, generated by a build step aggregating all metadata.json
files, imported statically by the app:

```json
{ "goblet-squat": { "frameCount": 6, "thumbnailFrame": 4 } }
```

Rules:

- The manifest is a **projection**: it carries only fields a component
  consumes. `credits`/`version` are authoring-only and never ship.
- `thumbnailFrame` is 1-indexed (matches `01.avif`). Default: the mid-rep
  frame (≈ N/2 + 1); hand-picked exceptions live in metadata.json.
- All asset paths are resolver-derived from `<id>` + convention. No paths
  are stored anywhere.
- Future `media.frameCaptions[]` (step-through captions, extracted from
  prompt.md frame descriptions) joins the manifest only when the UI reads it.

## App integration

- Single seam: `exerciseAsset(exerciseId, kind)` resolver reading the
  manifest synchronously. Missing entry → `null` → the existing asset-free
  layout. Assets never block training features.
- Components: `<ExerciseThumbnail>` (lazy `<img>`, `aspect-ratio` reserved)
  and `<FrameStepper>` — **CSS scroll-snap, no carousel library**
  (`IntersectionObserver` for caption/index sync). Embla Carousel is the
  designated fallback only if programmatic control (e.g. auto-advance during
  rest timer) becomes real; the component boundary makes that swap local.
- Surface order: Today-page thumbnail → Library detail step-through →
  list thumbnails → (later) rest-timer "next up" preview.
- Frames from `bench-side` / `floor-side` strips are landscape; standing
  strips are portrait. Components must handle both orientations.
- **Process rule (added after a production incident):** the generated
  manifest and the code that reads its schema land in the **same
  commit** — never split across a commit and an uncommitted working-tree
  change. Pre-push live verification runs against the committed tree,
  not the dev server's working tree, since the dev server happily reads
  files that were never committed and would pass a check the deployed
  build can't. The resolver still tolerates a stale/missing field either
  way (see src/lib/exerciseAsset.ts) — this rule is about not needing
  that tolerance in the first place.
- **Naming policy:** for any exercise that exists in the Library, the
  asset directory uses the **Library id verbatim**. The Library is the
  datasource of record; asset ids don't get to diverge from it going
  forward. `src/lib/exerciseAsset.ts`'s alias map exists only to bridge
  already-shipped legacy asset names generated before this policy, and a
  direct manifest hit under the Library id always wins over an alias —
  so a later batch landing a correctly-named asset supersedes its alias
  automatically, no code change. The map shrinks over time; it never
  grows as policy. New Library exercises get their asset generated under
  the Library id from day one, full stop.

## Offline / shipping

- Service-worker precache: runtime AVIFs for the **active program's
  exercises only** (~3–5 MB); the rest of the library lazy-loads.
- Precache excludes `*.png` and per-folder `metadata.json`. Source PNGs
  remain in the deploy for now; move to a `design-sources/` tree only if
  deploy size becomes a problem.

## Pilot (gate before converting the full library)

`goblet-squat`, `romanian-deadlift`, `bench-press`, `side-plank` — builds the
slicer + encoder + manifest step and validates: AVIF quality/size, landscape
frame handling (bench-press), the 2-frame hold case (side-plank), and
`thumbnailFrame` selection. Owner reviews before the remaining ~96 convert.

## Sequencing

1. Generation run completes → owner reviews all strips (rejects regenerate
   individually).
2. Pilot conversion (4 exercises) → owner review.
3. Full-library conversion.
4. Resolver + manifest build step + `<ExerciseThumbnail>` on Today page.
5. `<FrameStepper>` on the Library detail page.

## Relationship to prior docs

- `docs/ExerciseAssets.md` prescribed SVG-only shipping assets. Decision
  2026-07-19: the raster pipeline above is adopted for the current era; the
  SVG/motion eras remain future options. Its `metadata.json` coaching schema
  (muscles, cues, difficulty) is deferred — those fields are domain-owned.
- The folder-per-exercise, resolver-seam, and missing-asset-is-designed
  principles of that document remain in force.

## Parked decisions

- 8 near-duplicate consolidations (pec-deck / machine-chest-fly, pulldown
  grips, shrugs, etc.) — assets deferred, listed in the generation plan.
- Domain/asset id mismatches — **resolved via the resolver's alias map**
  (src/lib/exerciseAsset.ts), not by renaming either side: `single-leg-rdl`
  → `single-leg-romanian-deadlift` (this doc's original call-out);
  `barbell-squat` → `barbell-back-squat` and `bent-over-row` →
  `barbell-row` (approved on owner review once the asset side turned out
  to already cover them under those names).
- ~13 seeded exercises (band work, tempo/deficit/single-arm variants) have
  no prompts yet.
- `chatgpt-image-latest` requires org verification — optional alternative
  model, untested.
