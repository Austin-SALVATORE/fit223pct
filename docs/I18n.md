# Internationalization

**Status: designed, not yet implemented** (planned as Milestone 6 — see
Roadmap). This brief is the spec the implementation is reviewed against.
It amends the raw feature request where that request assumed surfaces
this app doesn't have.

## Goal

Three locales at launch: English (`en`, default and fallback), French
(`fr`), Simplified Chinese (`zh-CN`). Auto-detect on first launch
(`navigator.language`), manual switch in Settings, persisted in
`UserSettings`, live switch without restart. Adding a locale later means
adding one locale file — no code changes.

## What the generic spec missed about this app

1. **The domain layer speaks.** ~30 user-facing sentences are generated
   inside `domain/` (progression reasons, adjustment reasons, driver
   labels, trend phrases, stagnation evidence, coach insights). This is
   the product's core: every claim carries its reason. i18n therefore
   requires the app's single deepest refactor: **domain functions return
   message descriptors (key + params), never prose**. The UI layer
   translates. Example: `{ key: 'progression.deferLoad' }` instead of
   "You've earned more load, but not today…". Domain stays pure and
   testable (tests assert keys, not English); copy moves to locale
   files where it belongs.
2. **Seeded data is user-facing.** Exercise names, technique cues,
   prescription notes, program and session names live in Dexie, seeded
   at first run. Locale files can't reach them. Seed content moves to
   locale-keyed lookup by stable id (`exercise.goblet-squat.name`),
   resolved at render — stored records keep ids only. Program/session
   names authored per phase are content, translated with the phase.
3. **No Settings surface exists.** The language switcher needs the
   app's first Settings page — minimal: a quiet header or Today-footer
   entry, language selection only. Nothing else moves into it yet.
4. **Struck from the request** (surfaces that don't exist by design —
   local-first, single-user, no server): authentication, onboarding,
   profile, success notifications. "Recovery" in the request is a
   terminology error — see below.

## Invariants (review will check these)

- **Storage stays locale-free.** IndexedDB records never contain
  translated text — ids, signal keys, tier names, message keys only.
  (The M3 driver-keys fix established this; nothing may regress it.)
- **Terminology constraints carry into every locale.** "Readiness" —
  never wellness, recovery-score, or medical vocabulary — must hold in
  French and Chinese too. Each locale file starts with a small glossary
  (readiness tiers, RIR, the five signals) agreed before bulk
  translation; translations are natural, not literal, and fitness
  terms sound like a coach, not a manual.
- **Plurals via ICU messages, not ternaries.** The five hand-rolled
  `x === 1 ? 'set' : 'sets'` sites migrate; French pluralizes
  differently and Chinese doesn't pluralize at all.
- **Dates and numbers via `Intl`** with the active locale
  (`formatLongDate`, evidence dates, units). Units stay metric.
- **Fallback is English** for any missing key; a missing key in dev is
  a console error, never silent.

## Implementation shape

- **Library**: react-i18next (battle-tested, live switching, ICU via
  plugin, TS-typed keys) — confirm with a short eval at kickoff before
  committing.
- **Locale files**: `src/locales/{en,fr,zh-CN}/` split by module
  (`domain.json`, `today.json`, `workout.json`, `progress.json`,
  `library.json`, `checkin.json`, `common.json`, `seed.json`).
- **Order of work**: descriptor refactor in domain (with tests moved to
  keys) → extract feature-component strings → seed-content lookup →
  Settings page + detection/persistence → French → Chinese → docs on
  adding a locale.
- **`<html lang>`** updates with the locale; the a11y audit's focus
  and title handling already exists and must announce in the active
  language.

## Acceptance

Switching language updates every visible string without reload; all
three locales complete; English fallback proven by removing a key in
dev; stored data unchanged byte-for-byte before/after a language
switch; glossary reviewed by a native reader per locale before ship.

## Out of scope (deliberate)

RTL support (no RTL locale planned), per-locale content differences,
translating past data (there is none — storage is locale-free), server
locale negotiation (no server).
